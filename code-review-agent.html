the problem

Code review is a bottleneck

Developers wait. Reviewers miss patterns under time pressure. Security issues slip through when nobody checks every file. This is the gap CodeReviewAgent fills.

PAIN 01

Review lag

PRs sit open waiting for a reviewer who's busy. Merge delays compound, blocking downstream work.

PAIN 02

Inconsistent depth

Review quality varies by reviewer, time of day and PR size. Large PRs get less scrutiny than small ones.

PAIN 03

Security blind spots

SQL injection, exposed secrets and insecure patterns need a trained eye on every file — manual review rarely delivers that consistently.

03 / 14
built in-house

CodeReviewAgent — built here, owned here

Not a third-party plugin. A first-party agent built on Azure Functions and Azure AI Foundry, integrated directly into ADO — so every instruction, every prompt layer, and every model choice is ours.

Trigger layer

Azure DevOps

Listens for PR comment events. No new tooling for developers — they stay in the ADO PR they already have open.

Runtime layer

Azure Functions

Serverless, event-driven. A webhook fires on a matching comment and the function handles the review lifecycle end to end.

Intelligence layer

Azure AI Foundry

The model used for all review, security analysis, description and Q&A — model choice and version are controlled by the team.

05 / 14
end to end

From comment to review in seconds

💬

Developer comments on the PR

Any PR comment mentioning @CodeReviewAgent followed by a valid command is the trigger — no form, no UI, no new tab.

🔔

ADO fires a webhook

Azure DevOps emits a comment-created event to the registered Function endpoint. The function validates the payload and extracts the command.

POST /api/code-review-agent
🏗️

Azure Function assembles the prompt

It pulls the diff from ADO, identifies file types, and stacks the three prompt layers in order — base, then language (L1/L2), then repo instructions (L3).

🧠

Foundry model reviews the diff

The assembled prompt is sent to Azure AI Foundry. The model reads the code through the lens of all active instruction layers.

✅

Response posted back to the PR

The review, answer, security report, or description is posted as a new PR comment — appearing inline in the same conversation that triggered it.

prompt architecture — the core design

Three layers of instruction

Every review is assembled from three stacked prompt layers. Each one is additive — a later layer never replaces an earlier one, it extends it.

Base Layer — always active
Grounding & Safety
security · tone · output format · guardrails
Layer 1 — always active
All Languages
logic · naming · patterns · error handling
Layer 2 — file-type matched
Language-Specific
Review.sql.md · Review.cs.md · Review.ts.md · Review.tsx.md

Base layer — the floor

Applied to every single request regardless of command or language. Sets the model's role, tone, output format, safety boundaries, and non-negotiable security checks. Cannot be overridden by later layers.

Layer 1 — language-agnostic quality

Default review prompts for code quality, readability, naming conventions, error handling and architectural patterns — applicable to every supported language without distinction.

Layer 2 — language-specific depth

Adds targeted checks per file type. A .sql file gets SQL-specific injection checks, index and query plan guidance. A .ts file gets strict-mode, type-safety and async-pattern review. Multiple layers activate in a single PR if it touches multiple file types.

07 / 14
base layer

Grounding, safety & security — the non-negotiable floor

The base layer is a constant. It defines how the agent behaves across every command, every repo, every file type. Subsequent layers add onto it — they cannot remove from it.

base-layer.md — safety & guardrails

Model grounding. Instructs the model to act as a code reviewer, not an assistant — focused, factual, and specific to the diff only.

No hallucinated references. The model may only reference what is visible in the changed files and the repository's declared dependencies.

Confidentiality. The agent never echoes back secrets, environment variables or credential-like strings it encounters in the diff.

Non-opinionated tone. Findings are reported as observations with rationale, not mandates — the developer retains decision authority.

base-layer.md — always-on security

Secret detection. Flags any string that looks like an API key, token, password or connection string — even in comments.

Injection surface. Checks every string-interpolation pattern that touches an external system (SQL, shell, XML, LDAP).

Auth and access. Flags missing authorisation checks, direct object references, and overly permissive roles.

Dependency flags. Notes newly added packages and recommends checking their vulnerability status.

layer 1 & layer 2

Language coverage — broad by default, deep when needed

Layer 1 fires for every file. Layer 2 is matched per file extension — one review pass, multiple layers active simultaneously if the PR touches multiple languages.

Layer 1 — default.md · all languages

Code clarity and naming conventions. Logic correctness and edge-case coverage. Error handling completeness. Dead code and redundant imports. Magic numbers and undocumented assumptions. Test coverage of new behaviour.

C#TypeScriptJavaScriptSQLPythonPowerShellBicepYAMLJSON+ any
Layer 2 — file-matched specialist prompts

Review.sql.md — parameterised queries, N+1 patterns, index usage, transaction scope, stored-proc risks.

Review.cs.md — nullable safety, async/await correctness, LINQ efficiency, DI pattern adherence, exception handling.

Review.ts.md — strict-mode types, any/unknown discipline, Promise chains, module boundaries.

Review.tsx.md — component decomposition, hook rules, memoisation, accessibility, key stability.

layer 3 — optional, per-repo

Customised per repository

Any repo can pass its own review instructions by placing a file in its root. The agent picks them up automatically — no deployment or configuration change required.

📁 repo-root / .github / instructions
# repo-specific instructions for CodeReviewAgent # picked up automatically when this repo is reviewed focus_areas: - Performance: flag any DB call inside a loop - Naming: follow our domain-model glossary - Auth: all API endpoints must have [Authorize] suppress: - line-length warnings (we use 140-char width) style: - Use our internal terminology: "Member" not "User" - Reference ADO work items where known
01

Zero deployment needed. The agent checks for the file at review time. Add or edit it and the next review uses it — no PR to the agent repo required.

02

Additive, not overriding. Layer 3 instructions extend the base and language layers — they cannot suppress the base layer's safety and security checks.

03

Context-aware terminology. If your domain uses specific names for entities, components or patterns, put them here — the agent will use and recognise them.

04

Team-owned, version-controlled. The file lives in the repo, so changes are reviewed and tracked like any other configuration decision.

outcomes & guardrails

What it changes — and where humans stay in the loop

speed

Instant first feedback

A developer gets structured feedback within seconds of posting the command — no waiting for a reviewer's calendar to clear.

consistency

Same depth on every PR

The base and language layers don't have bad days. Every file gets the same security and quality pass, regardless of PR size or reviewer availability.

security

Security on every diff

The base layer's always-on security check means no file reaches main without at least a first-pass scan for common vulnerabilities.

adoption

Zero new tooling for devs

Developers stay in ADO. One comment is the entire interface — no new dashboard, no plugin, no context switch.

GUARDRAIL 01

Advisory only

The agent cannot merge, approve, or block PRs. All its output is a comment. The pull-request decision always belongs to a human reviewer.

GUARDRAIL 02

No secrets echoed

The base layer instructs the model to never repeat credential-like strings from the diff, even in its analysis output.

GUARDRAIL 03

Diff-scoped only

The model is given only the changed files, not the entire repository history — limiting both scope and cost per review call.

GUARDRAIL 04

L3 cannot override safety

Repo-specific instructions are additive. No Layer 3 instruction can suppress base-layer security checks or change the agent's fundamental output format.
