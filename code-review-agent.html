Seven validation gates before any message is enqueued

Every inbound request passes a strict sequential pipeline. The function returns HTTP 202 at step 7 — ADO never times out waiting for review results.

01PASSKill switch — IsEnabled check → HTTP 200 silent no-op if false
02SECWebhook secret — CryptographicOperations.FixedTimeEquals() on X-Webhook-Secret header → HTTP 401 on mismatch
03SZPayload size guard — rejects bodies > 512 KB (checks Content-Length + post-read body length) → HTTP 413
04JSONDeserialize AdoWebhookEvent — extract EventType, PullRequestId, Repository, SourceCommit
05MEML1 dedup — IMemoryCache sliding TTL 10 min, capped at 2 000 entries (evicts 200 oldest on cap) → HTTP 200 if seen
06TBLL2 dedup — TableWebhookDedupService.TryClaimEventAsync() Table Storage cross-instance → HTTP 200 if already claimed
07ROUTEEvent routing — created / updated / comment-event → build ReviewQueueMessage → enqueue → HTTP 202
Multi-output pattern — .NET isolated worker v4

The function uses a WebhookOutput wrapper class with [QueueOutput] on a property — required in isolated-worker v4 to simultaneously return an HttpResponseData and enqueue a message. Do not refactor to method-level attributes — it breaks HTTP response delivery.

Event type routing (case-insensitive .ToLowerInvariant())

git.pullrequest.created → FullReview auto-trigger

git.pullrequest.updated → Incremental review on new source commit; vote/status changes silently filtered

ms.vss-code.git-pullrequest-comment-event → On-demand command, only when comment contains BotMentionHandle

Four defence gates before any AI call is made

The processor re-validates business rules independently of the receiver — defence in depth. On-demand @mention commands post an immediate ⏳ On it… acknowledgement before any AI work begins.

GATE 1

IsEnabled kill-switch

Global on/off. Message is completed (not retried) — prevents queue build-up during deliberate maintenance shutdown.

GATE 2

AutoReviewEnabled

Skips auto-triggered events independently of on-demand commands. Disabling auto-review never blocks @mention commands.

GATE 3

Repository allow-list

Optional list of repository names to restrict reviews. Empty = all repos allowed. Checked against the queue message.

GATE 4

PR status check

Skips completed or abandoned PRs for auto events. On-demand @mention commands bypass this — developer explicitly requested review.

LCS unified diff — 80–90% token reduction on edits

For edited files, the agent computes a compact Myers-LCS unified diff instead of sending before + after content. A 200-line file with 10 changed lines uses 90% fewer tokens.

DiffHelper.cs — Myers LCS unified diff algorithm

Edit files: ComputeUnifiedDiff(before, after, contextLines=3) — LCS backtrace, outputs + added / - removed / space context. Unchanged regions →   ... N unchanged line(s) skip marker.

New files: Full content sent — every line is new, no diff needed.

Deleted files: Skipped entirely — no value in reviewing removed code.

Pre-existing detection: GetChangedNewFileLineNumbers() returns the set of 1-based line numbers of + operations. Post-AI pass: findings on unchanged lines tagged pre-existing at original severity.

Limits (configurable): MaxFilesPerReview = 20 · MaxLinesPerFile = 500

Three phases of intelligent re-review on every push

Each new commit push triggers an incremental review. Previously addressed findings are not re-raised. Developer pushback across iterations is respected.

Phase 1 — Thread-aware inline comment management

Before posting, ReviewProcessor fetches all existing PR threads. Each new finding is matched to an existing agent thread by: file path (normalised) + category (ADO thread property CodeReviewBot.Category) + line proximity ±5 lines.

Match found + active → reply “Still present” (no duplicate thread). Active thread unmatched → auto-resolved “no longer present” (status set to fixed).

Phase 2 — Incremental iteration diff

On git.pullrequest.updated: ReviewStateService reads last reviewed iteration ID from BotReviewState Table Storage. Calls GET /iterations/{target}/changes?compareTo={base} — only files changed since last review.

Example savings: PR with 20 files where 3 changed → ~85% input token reduction. Falls back to full diff if incremental returns 0 files.

Phase 3 — Conversation context injection

ConversationContextBuilder.BuildPriorReviewContext() builds a compact ## Prior Review Context section injected into the AI user message. Capped at 10,000 characters (~2,500 tokens).

Contains: active agent findings (severity + location) → resolved findings → developer replies (up to 2 per thread). AI is instructed: do not re-raise [RESOLVED], [WON’T FIX], or [BY DESIGN] threads.

Pre-existing issue detection — post-processing pass

After the AI returns findings, AiReviewService.AnnotateFindingsWithDiffContext() runs. For each finding: if the reported line is not in the changed-lines set → IsInDiff = false.

Severity is never changed — a pre-existing CRITICAL stays 🔴 CRITICAL. Tagged “pre-existing — not introduced by this PR” in summary + inline thread body.

Webhook secret validation

Every inbound request validates X-Webhook-Secret header against Bot--WebhookSecret from Key Vault. Comparison uses CryptographicOperations.FixedTimeEquals() to prevent timing attacks. Missing or incorrect secret → HTTP 401. Recommended for all non-local environments.

PAT-free option — managed identity only

Leave Bot--AdoPatToken empty in Key Vault. Grant the Function App identity Basic access in ADO org with Contribute to pull requests. AdoService acquires AAD Bearer tokens lazily via DefaultAzureCredential with a SemaphoreSlim(1,1) lock and 5-minute TTL refresh buffer. Zero PAT rotation forever.
