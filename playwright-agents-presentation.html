Why Playwright

A modern automation library that has grown an agentic layer on top of it — and now a second, lighter-weight interface built specifically for agents.

One API, three engines

Chromium, Firefox and WebKit driven through the same API, so a single test suite covers real cross-browser behaviour.

Auto-waiting, web-first assertions

Actions wait for elements to be actionable and assertions retry until they pass, removing most manual sleep() calls that cause flake.

Built-in Test Agents

Planner, Generator and Healer connect an LLM to a real browser session, scaffolded straight from the Playwright CLI.

playwright-cli for agent loops

A separate, token-efficient @playwright/cli package built for high-iteration agent work — this is what our Healer runs on.

Planner starts from Azure DevOps, not a blank page

Instead of a free-text prompt, Planner is pointed at an ADO Test Plan / Suite — the test case inventory the team already maintains becomes the source of truth.

1

Fetch from ADO

Planner connects to the Azure DevOps Test Case / Suite API, pulling title, steps, expected results and tags for each case in the suite.

2

Ground it in APT

Each ADO step is replayed against the live app — the same explore-and-verify pattern as a prompt-driven plan, just driven by ADO content instead of free text.

3

Plan with traceability

The resulting plan keeps a front-matter link back to the ADO test case ID, so every test we generate can be traced to the requirement that justified it.

specs/937578-risks-plan.md → ado_test_case: 4821
4

Flag drift, don't silently fix it

A step that no longer matches the app is called out in the plan rather than quietly rewritten — a human decides whether the app or the ADO case is out of date.

Why Healer runs on playwright-cli

Healing is the most iteration-heavy phase — dozens of small snapshot, click and inspect calls per failing test. @playwright/cli keeps that loop cheap and grounded.

$ playwright-cli open https://apt.internal/checkout
$ playwright-cli snapshot
→ saved .playwright-cli/page-2026-06-21T10-02Z.yml
$ playwright-cli click e58
$ playwright-cli screenshot
$ playwright-cli network
$ playwright-cli console
# reads the snapshot only when it needs the detail
01

Saved to disk, not streamed. A snapshot lands as a YAML file; Healer reads it only when it actually needs the detail, instead of a full accessibility tree on every turn.

02

Ref-based, not selector-guessed. Elements are addressed by stable refs captured at snapshot time — fewer fragile, freshly-guessed CSS chains.

03

Daemon architecture. One persistent browser process backs the whole repair loop — no relaunch cost between each small check.

04

Same Playwright underneath. Still Chromium/Firefox/WebKit — Playwright's reliability without the full tool-schema overhead.

Project layout

Agent output isn't the end state — it's a contributor. ADO-sourced plans and generated tests land inside the same Page Object Model every hand-written test already follows.

project/
├─ agents/ # planner · generator · healer definitions Tier A
├─ specs/ # Markdown plans, linked to ADO test case IDs Tier B
├─ ado-cache/ # TC-937578.json — source ADO case, for reference Tier C
├─ tests/ Tier C
│  ├─ seed.spec.ts # baseline state + auth, used by all agents
│  └─ <feature>-tc<id>.spec.ts # generated + healed tests
├─ pages/ Tier D
│  ├─ base.page.ts # shared base POM for APT
│  ├─ login.page.ts
│  └─ <feature>.page.ts # locators + actions per screen
├─ helpers/ # shared utilities, test data builders Tier D
├─ auth/ Tier D
│  └─ storageState.json # persisted, authenticated session
├─ apt-learnings.md # auto-updated after every session
└─ playwright.config.ts

.playwright-cli/ is Healer's runtime snapshot cache — gitignored, regenerated every run.

Base Page Object Model for APT

A shared base class plus one Page per screen — the contract every agent and every human-written test calls into.

pages/login.page.ts
export class LoginPage extends BasePage {
  constructor(private page: Page) {
    super(page);
  }

  private email    = () => this.page.getByLabel('Email');
  private password = () => this.page.getByLabel('Password');
  private submit   = () =>
    this.page.getByRole('button', { name: 'Sign in' });

  async signIn(user: string, pass: string) {
    await this.email().fill(user);
    await this.password().fill(pass);
    await this.submit().click();
  }
}
01

One place to fix UI changes. A renamed field is a one-line edit inside the Page class, not a hunt across every spec file.

02

Tests read like requirements. await loginPage.signIn(user, pass) states intent without exposing implementation.

03

Healer repairs locators, not logic. playwright-cli patches stay scoped inside the Page class — the test's meaning never drifts.

04

Generator reuses what exists. When a flow already has a Page Object, it calls into it instead of writing fresh locators from scratch.

Authenticate once, reuse everywhere

A single real login, captured once, becomes the starting point for every test and every agent run.

1

One real sign-in

A setup project — or the seed test itself — logs into APT through the actual UI, exactly once per run.

2

Session is persisted

Cookies and local storage are written to auth/storageState.json via Playwright's built-in storage state capture.

3

Every project reuses it

Test projects in playwright.config.ts point use: { storageState } at the saved file, so suites launch already signed in.

4

Agents inherit the same session

Planner runs the seed test before exploring, so it starts authenticated too — credentials live in the helper, never in an agent prompt.

Not all failures are the same — triage first

Sending every failure straight to Healer wastes tokens. Classifying the failure type first lets us route it to the right response.

TYPE 01 — FLAKY

Timeout & locator randomness

Element not found, timeout exceeded, or a race with a loading state. Often intermittent — the same test passes on retry. Usually safe for Healer to address.

✓ Healer auto-fixes
TYPE 02 — DATA SETUP

Missing or incorrect test data

A record the test expects doesn't exist, has the wrong state, or was mutated by a previous run. Healer can flag it, but fixing it needs a data-setup strategy.

→ seed data / isolated accounts
TYPE 03 — INFRA / THROTTLE

Server busy or too many requests

HTTP 429, 503, or slow response under load. The app code is fine — the environment isn't ready. Retry with back-off, or move the test to off-peak CI scheduling.

→ retry strategy, not a code fix
TYPE 04 — GENUINE REGRESSION

The app broke something real

A feature changed, a contract was violated, or a business rule no longer holds. Healer must not silently patch these — flag for human review and raise a defect.

⚠ Human decision required

Phase 1

Single skill

One Claude Code skill, one prompt template. Fast to set up, quick to hit its ceiling — no memory between sessions, no APT-specific context.

Phase 2

Three agents

Planner, Generator, Healer — each with its own tool set and system prompt. Clear separation of concern, ADO sourcing wired in, playwright-cli powering Healer.

Living knowledge — apt-learnings.md

Agent instructions and apt-learnings.md grow after every session. Patterns, anti-patterns, and APT-specific tips are captured automatically and read back on the next run.

Healer challenges & human-in-the-loop

Healer is powerful but not free — it takes time, spends tokens, and can over-engineer. Knowing when to steer it matters.

01
Token & time cost

Each playwright-cli loop iteration consumes context. A deep repair session on a hard failure can burn significant AI credits. Set a max-iteration budget per run.

02
Over-engineering simple fixes

Without guidance, Healer may chase locator chains when the real cause is test data or test ordering. Tell it the failure type upfront to skip the wrong search.

03
Genuine regressions need human eyes

If Healer "fixes" a test by weakening an assertion to make it pass, that's a false positive. Review all healed diffs before merging.

04
Modules not yet covered

Multi-user, multi-engagement, conflict detection, and questionnaire scenarios are on the roadmap but not yet generated — human-authored tests cover these today.

Where humans stay in the loop

Agents accelerate the work; they don't own the judgment calls.

01

Review every generated or healed diff

Plans, generated tests and Healer patches all go through normal code review before merging — same bar as human-written code.

02

Drift gets flagged, not silently resolved

When an ADO step no longer matches the app, Planner calls it out — a person decides which side is wrong.

03

Genuine regressions are never auto-healed

Healer patches locators and data setup. It marks tests skipped with a reason rather than weakening assertions to force a pass.

04

Steer Healer before it goes deep

If you know the failure type, say so in the prompt. A short context note can save dozens of costly repair-loop turns.

05

Agents don't replace test strategy

What to test, and why, stays a team decision — the pipeline accelerates execution, not judgment.

06

Credentials never enter a prompt

Auth relies on storage state captured once from the seed test. Credentials live in helpers, not in agent context.

What's not covered yet

Several APT modules are still being implemented or are planned for the next phase of test generation.

multi-engagement

Workflows that span multiple stages across separate logins or sessions — not yet planned in the agent pipeline.

multi-user

Scenarios where two users interact with the same record simultaneously — requires parallel browser contexts.

conflict detection

Concurrent edits, lock warnings, and merge-conflict flows — these tests exist today but are hand-authored.

questionnaire

Long forms with conditional logic and validation chains — Generator needs branching plan support before these can be automated.
